# プロジェクトルール

## デザインと作業範囲
- ユーザーがリクエストしたこと「だけ」を「正確に」実装する
- 余計な機能、コンポーネント、デザインの追加は一切しない
- 指示が曖昧な場合は、最もシンプルで妥当な解釈を選ぶ

## 作業の進め方
- **複数の指示がある場合、または1つの指示でも**、必ず以下の流れを踏む:
  1. **計画**: 何をするか明確にする
  2. **確認**: ユーザーの承認を得る
  3. **実行**: 承認後に実装
- 重い作業は小分けにして、こまめに確認を取る

## 不確実性・曖昧さへの対処
- 質問が曖昧な場合は、1〜3個の具体的な質問で確認するか、2〜3個の解釈の可能性を提示する
- 不確かな場合は、「提供された情報によると…」のような断定的でない表現を使う
- リスクの高い分野では、「根拠のない情報はないか」「断定しすぎていないか」を自己チェックする

## 出力の長さ
- 通常の回答は3〜6文、または5つ以下の箇条書きで
- 「はい/いいえ」で答えられる簡単な質問には、2文以内で
- 複雑な作業は「概要」「変更点」「リスク」「次のステップ」などで整理して報告
- 長い文章ではなく、箇条書きや短いセクションを好む

## コミットメッセージ
- **日本語**で自然な表現を使う
- 例: 「修正: ボタンがクリックできない問題を解消」
- 例: 「機能追加: メモ入力時にログを分割する機能」

## コメント・説明
- ユーザーへの回答は**日本語**で行う
- コード内のコメントは英語でも日本語でもOK（既存のスタイルに合わせる）

## 変更後の確認
- 変更が完了したら、必ず以下を確認する:
  - **チェックポイント**: `git stash` で一時保存（後でロールバック可能）
  - **コミット**: `git commit` で正式に記録
- どちらにするかユーザーに確認してから実行する

## その他
- エラーが発生した場合は原因を調査してから修正する
- ブラウザテストが必要な場合は積極的に実行する

## 単一HTMLファイルの更新
コード変更後、以下のコマンドで `wheel-tracker-v3-standalone.html` を最新版に更新する:
```powershell
// turbo
npm run build
// turbo
Copy-Item "dist\index.html" -Destination "wheel-tracker-v3-standalone.html" -Force
```
