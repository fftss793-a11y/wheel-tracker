# プロジェクト作業ルール

## コミットメッセージ
- **日本語**で自然な表現を使う
- 例: 「修正: ボタンがクリックできない問題を解消」
- 例: 「機能追加: メモ入力時にログを分割する機能」

## コメント・説明
- ユーザーへの回答は**日本語**で行う
- コード内のコメントは英語でも日本語でもOK（既存のスタイルに合わせる）

## 作業の進め方
- **計画・確認・実行の徹底**: 必ず以下の流れを踏み、各ステップ間で明示的なユーザーの承認を待つ：
  1. **計画**: 実装方針を `implementation_plan.md` にまとめ、`notify_user` で提示する。
  2. **確認**: ユーザーから「OK」「進めて」等の承認を得る。**承認を得るまで、一切のコード編集（実行）を行わない。**
  3. **実行**: 承認された方針に沿って実装する。
- **リファクタリングの制限**: 「余計なことをしない」原則に基づき、共通コンポーネント化やコード整理が必要な場合は、必ず「計画」段階で提案し、承認を得ること。独断でのリファクタリングは避ける。
- **透明性の担保**: 毎ターン `task_boundary` を使用し、現在どのフェーズ（計画中・実行中）にいるかをユーザーに明確に伝える。

## 変更後の確認
- 変更が完了したら、必ず以下を確認する:
  - **チェックポイント**: `git stash` で一時保存（後でロールバック可能）
  - **コミット**: `git commit` で正式に記録
- どちらにするかユーザーに確認してから実行する

## その他
- エラーが発生した場合は原因を調査してから修正する
- ブラウザテストが必要な場合は積極的に実行する

## 単一HTMLファイルの更新
コード変更後、以下のコマンドで `wheel-tracker-v3-standalone.html` を最新版に更新する:
```powershell
// turbo
npm run build
// turbo
Copy-Item "dist\index.html" -Destination "wheel-tracker-v3-standalone.html" -Force
```
